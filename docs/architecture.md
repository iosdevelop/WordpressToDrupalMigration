# Architecture

## Scope

This prototype demonstrates a credible migration slice from WordPress to Drupal 10, with explicit structure for scaling from a sample set to a full migration of 275 legacy pages into about 125 Drupal destinations.

## Source WordPress Fixtures

WordPress fixtures are generated by `scripts/create-mock-pages.sh` using WP-CLI in DDEV.

Key properties:

- deterministic slugs and titles
- 40+ pages to represent realistic migration volume
- HTML body content and internal legacy links
- post meta for migration metadata continuity:
  - `legacy_url`
  - `seo_title`
  - `seo_description`
- edge cases included for prototype discussion:
  - nested pages
  - missing meta description
  - long-title content
  - basic image markup in HTML body

## Migration Flow

1. Generate WordPress fixtures in `/wordpress`.
2. Export fixtures to `data/wordpress-pages.csv` using `scripts/export-wordpress-pages.sh`.
3. Generate mapping artifacts with `scripts/generate-old-to-new-map.sh`:
   - `data/old-to-new-map.csv`
   - `data/sample-redirects.csv`
4. Copy migration CSVs into Drupal module-local data directory.
5. Run Drupal migrations (`wp_pages`, `wp_redirects`) with Drush.

## Drupal Custom Module Role

Module: `drupal/web/modules/custom/wp_drupal_prototype_migrate`

Responsibilities:

- owns migration YAML definitions
- owns helper service and process plugins for URL normalization
- creates destination content type/fields in install hook
- stores migration input CSV snapshots under `data/`

Destination model for prototype:

- content type: `wp_migrated_page`
- mapped fields:
  - `title`
  - `body`
  - `path alias` (`/prototype/<slug>`)
  - `field_legacy_url`
  - `field_seo_title`
  - `field_seo_description`

## Redirect Strategy

Prototype redirect strategy is CSV-driven and migration-imported.

- Source: `sample-redirects.csv` with `legacy_url`, `redirect_to`, `status_code`
- Migration: `migrations/wp_redirects.yml`
- Process plugins convert:
  - full legacy URL -> redirect source path
  - target path/URL -> `internal:` redirect URI

The prototype intentionally uses a consolidation pattern where multiple old URLs can map to fewer canonical Drupal paths.

## Metadata Preservation Strategy

Current prototype preserves key continuity metadata:

- legacy URL reference stored on destination node
- SEO title and description persisted to dedicated prototype fields
- redirect intent modeled separately from page migration for clean responsibilities

This is enough to demonstrate migration intent and continuity without claiming full production SEO parity.

## QA Strategy for Later Phases

Recommended next-phase QA controls:

- field-level parity report (source vs destination)
- alias uniqueness and collision checks
- redirect validation tests (status code + destination correctness)
- spot checks for broken internal links in migrated body HTML
- migration idempotency checks in CI

## Production Readiness Additions

To evolve this into production migration tooling:

1. Replace heuristic section mapping with governed canonical mapping matrix.
2. Add media migration pipeline and file integrity checks.
3. Add taxonomy/user/reference migrations and dependency ordering.
4. Add workflow/state mapping and editorial review loops.
5. Add dry-run environments, automated reconciliation reports, and rollback playbooks.
6. Add observability for migration throughput and error rates.
